## The core Gecko loop

Gecko is easiest to adopt when you treat it as a repeatable loop:

1. Connect a code source.
2. Run a scan.
3. Review the findings.
4. Fix what matters.
5. Rescan and confirm the result.

That loop is the product. Everything else in Gecko exists to make it faster,
clearer, or easier to scale across a team.

## What a new team should do first

<Steps>
<Step title="Connect your code source">
  Start with GitHub or GitLab if the repository is private and already part of
  your development workflow.

  If you are evaluating Gecko on a public project, use a public OSS repository
  URL instead.

  If you need a quick point-in-time review of code that is not in a provider,
  upload a ZIP.
</Step>

<Step title="Run a baseline scan">
  Use the default branch first.

  A baseline scan gives you the initial security picture for the repository and
  makes later re-scans easier to compare.
</Step>

<Step title="Triage the findings as a team">
  Review the first scan with someone who knows the codebase and someone who can
  decide what needs to be fixed now.

  Do not optimize for clearing everything in one pass. Optimize for creating a
  clean, repeatable workflow.
</Step>

<Step title="Connect workflow integrations">
  After the scan loop is working, connect ticketing, notifications, and SSO so
  Gecko fits the rest of your engineering process.
</Step>
</Steps>

## What to expect from your first week

<AccordionGroup>
  <Accordion title="You will find some issues worth fixing immediately">
    That is normal. Start with issues that have clear exploit paths, sensitive
    data exposure, auth bypass risk, or obvious remote impact.
  </Accordion>
  <Accordion title="Some findings will need validation">
    Gecko gives you evidence and remediation guidance, but your team still owns
    final judgment. Use code context and runtime knowledge to confirm impact.
  </Accordion>
  <Accordion title="The workflow gets better after the first baseline">
    Once you know what your initial result set looks like, it becomes much
    easier to track change, verify fixes, and connect issue-management tools.
  </Accordion>
  <Accordion title="SSO and provisioning are not the first priority">
    Start with code, scans, and remediation. Add SSO and SCIM when the team is
    ready to formalize access.
  </Accordion>
</AccordionGroup>

## Run scans, review findings, and fix issues

<Steps>
<Step title="Run scans on a steady cadence">
  Start with baseline scans on the default branch, then move to repeat scans
  that match how the team ships code.
</Step>

<Step title="Review findings with context">
  Use the finding details to understand exploitability, code context, and the
  safest remediation path before you assign work.
</Step>

<Step title="Fix the highest-value issues first">
  Prioritize findings that affect auth, secret handling, remote execution,
  external network access, or broad data exposure.
</Step>

<Step title="Rescan after remediation">
  The re-scan is what proves the issue is actually gone and that the codebase is
  in a better state than before.
</Step>
</Steps>
